# WeekFlow — план разработки первой версии

## 1. Рекомендуемый технологический стек

### Основной стек

- **React** — для интерфейса, состоящего из взаимосвязанных представлений недели, задач и форм редактирования.
- **TypeScript** — для явной модели проекта и задачи, безопасной работы с датами и снижения количества ошибок при развитии приложения.
- **Vite** — для простой и быстрой локальной разработки и сборки одностраничного приложения без сложного сервера.
- **Обычный CSS с CSS Modules** — для изолированных стилей компонентов без подключения тяжелой дизайн-системы. Общие дизайн-токены можно хранить в CSS-переменных.
- **date-fns** — для вычисления границ недели, сравнения дат и форматирования на русском языке. Даты задач внутри данных хранятся как календарные строки `YYYY-MM-DD`, чтобы не возникали сдвиги из-за часовых поясов.

### Проверка качества

- **Vitest и React Testing Library** — для проверки логики дат, хранилища и ключевых пользовательских действий.
- **Playwright** — для нескольких сквозных сценариев на размерах экрана Mac и телефона.
- **ESLint и Prettier** — для единообразия и базовой проверки исходного кода.

Отдельный менеджер состояния в первой версии не нужен: достаточно React Context в сочетании с reducer и отдельным слоем доступа к `localStorage`. Подключение Redux, серверной базы или UI-фреймворка усложнит MVP без доказанной пользы.

Зависимости устанавливаются только после утверждения этого плана.

## 2. Структура приложения и основные экраны

### Главный экран

На главном экране постоянно присутствуют только три ключевые зоны:

1. **Вся неделя.** Семь дней, загрузка, однодневные задачи и непрерывные полосы многодневных задач.
2. **Сегодня.** Компактный список задач текущего дня, включая задачи, чей период охватывает сегодня.
3. **Быстрое добавление.** Поле, в котором достаточно ввести текст. Без выбранной даты запись сохраняется во «Входящие»; детали можно раскрыть и заполнить вручную.

### Дополнительные представления

Они открываются поверх главного экрана или как отдельное компактное представление и не создают постоянные дополнительные зоны:

- **Входящие** — список задач без дат со счетчиком, разбором и назначением проекта, периода и приоритета;
- **редактор задачи** — боковая панель или модальное окно для создания и изменения деталей;
- **проекты** — компактное управление названием, цветом и архивным состоянием проектов;
- **подтверждение удаления** — короткий диалог для защиты от случайной потери данных.

### Предлагаемая структура модулей

- `app` — запуск приложения, общее состояние и маршрутизация представлений;
- `features/week` — недельная сетка, навигация и расчет отображаемых задач;
- `features/today` — выборка задач текущего дня;
- `features/quick-add` — быстрый ввод без ИИ;
- `features/inbox` — список и разбор задач без дат;
- `features/tasks` — карточка, полоса, редактор и операции над задачами;
- `features/projects` — управление проектами и их цветами;
- `storage` — чтение, проверка, миграция и запись локальных данных;
- `domain` — типы, инварианты и чистые функции для дат и задач;
- `demo` — только вымышленные начальные данные;
- `shared` — небольшие переиспользуемые элементы интерфейса и стили.

Это ориентир для этапа реализации, а не указание создавать структуру сейчас.

## 3. Модель данных

### Проект

```ts
type Project = {
  id: string;
  name: string;
  color: string;
  isArchived: boolean;
  createdAt: string;
  updatedAt: string;
};
```

Правила:

- `id` уникален и не зависит от названия;
- `color` принадлежит проекту и одинаково используется во всех представлениях;
- личные дела и обучение представлены обычными проектами;
- архивный проект сохраняется для старых задач, но не предлагается по умолчанию при создании новых.

### Задача

```ts
type Task = {
  id: string;
  title: string;
  note: string;
  projectId: string | null;
  priority: "none" | "low" | "medium" | "high";
  status: "active" | "completed";
  startDate: string | null;
  endDate: string | null;
  createdAt: string;
  updatedAt: string;
  completedAt: string | null;
};
```

Правила:

- даты хранятся как `YYYY-MM-DD` по локальному календарю пользователя;
- у запланированной задачи заполнены обе даты;
- у однодневной задачи `startDate === endDate`;
- у многодневной задачи `startDate < endDate`;
- `endDate` не может быть раньше `startDate`;
- у входящей записи обе даты равны `null`;
- состояние, в котором заполнена только одна дата, недопустимо;
- до разбора входящей записи `projectId` может быть `null`, а приоритет — `none`;
- удаление проекта требует либо переназначить его задачи, либо сохранить проект в архиве.

Отдельные модели для входящей записи, идеи и категории не создаются.

## 4. Этапы разработки и критерии приемки

### Этап 0. Зафиксировать требования и правила данных

Работы:

- утвердить `concept.md` и `plan.md`;
- описать инварианты дат, правила просрочки, загрузки дня и завершения задач;
- согласовать простую схему интерфейса для Mac и телефона;
- подготовить перечень полностью вымышленных демонстрационных данных.

Критерии приемки:

- нет противоречий между концепцией, моделью данных и сценариями;
- для однодневной, многодневной и входящей задачи определено однозначное поведение;
- на главном экране предусмотрены ровно три ключевые зоны;
- ни один пример не содержит реальных данных.

### Этап 1. Создать каркас и слой данных

Работы:

- создать React-проект после отдельного подтверждения;
- настроить TypeScript, стили и проверки качества;
- реализовать типы, чистые функции дат и слой `localStorage`;
- добавить безопасную загрузку, проверку версии схемы и вымышленные демоданные.

Критерии приемки:

- приложение запускается и собирается локально;
- данные корректно читаются и записываются после перезагрузки;
- поврежденные или несовместимые данные не приводят к пустому или сломанному экрану;
- тесты покрывают инварианты дат и базовые операции хранилища;
- в репозитории отсутствуют реальные пользовательские данные.

### Этап 2. Реализовать недельный обзор

Работы:

- отобразить текущую неделю и навигацию между неделями;
- показать цвета проектов, отдельные метки приоритета, состояния завершения и просрочки;
- реализовать однодневные карточки и одну непрерывную полосу для многодневной задачи;
- добавить понятную индикацию загрузки дней.

Критерии приемки:

- вся неделя читается на одном экране Mac без перехода по отдельным дням;
- одна многодневная задача визуально остается одной сущностью и пересекает нужные дни;
- полосы корректно обрезаются на границах отображаемой недели и продолжаются на соседней;
- цвет проекта последователен, а приоритет распознается независимо от цвета;
- просроченность рассчитывается относительно локальной календарной даты;
- пользователь за 10 секунд определяет загрузку, проекты, просрочку и высокий приоритет на демоданных.

### Этап 3. Добавить «Сегодня» и операции с задачами

Работы:

- вывести задачи, период которых включает текущую дату;
- добавить создание, редактирование, завершение, перенос и удаление задачи;
- реализовать ручной выбор проекта, приоритета, даты начала и даты завершения;
- добавить подтверждение удаления и проверку корректности периода.

Критерии приемки:

- «Сегодня» содержит однодневные и многодневные задачи текущего дня без дублей;
- все операции сразу отражаются и в неделе, и в блоке «Сегодня»;
- невозможно сохранить пустое название или период с датой завершения раньше даты начала;
- после перезагрузки результат операций сохраняется;
- основные действия доступны с клавиатуры и сенсорного экрана.

### Этап 4. Реализовать быстрое добавление и «Входящие»

Работы:

- добавить сохранение текста одним коротким действием;
- сохранять запись без дат как обычную задачу во «Входящих»;
- сделать счетчик и отдельное компактное представление входящих записей;
- реализовать разбор записи с ручным назначением проекта, периода и приоритета;
- дать возможность сразу раскрыть дополнительные поля при быстром добавлении.

Критерии приемки:

- мысль можно сохранить менее чем за 10 секунд, введя только текст;
- для быстрого сохранения не требуются дата, проект и приоритет;
- входящая запись не появляется в недельной сетке до назначения дат;
- после назначения дат та же запись исчезает из входящих и появляется в неделе без смены `id`;
- функциональность не использует ИИ и не отправляет текст во внешние сервисы.

### Этап 5. Добавить управление проектами и визуальную целостность

Работы:

- добавить создание, изменение и архивирование проектов;
- ограничить палитру различимыми цветами с достаточным контрастом;
- проверить единое использование цвета проекта во всех представлениях;
- обеспечить отдельное обозначение приоритета формой, текстом или значком.

Критерии приемки:

- изменение цвета проекта обновляет все связанные задачи;
- цвет никогда не является единственным признаком приоритета или состояния;
- архивирование не разрушает связь с существующими задачами;
- интерфейс остается понятным при нарушении цветового восприятия и в режиме повышенного контраста.

### Этап 6. Адаптировать интерфейс и провести финальную проверку

Работы:

- настроить компоновку для Mac и телефона;
- проверить клавиатурную навигацию, фокус, подписи и размеры сенсорных целей;
- добавить состояния пустых списков, ошибок хранилища и подтверждения действий;
- выполнить ручные и автоматические проверки ключевых сценариев;
- провести аудит приватности перед любой публикацией.

Критерии приемки:

- интерфейс проверен как минимум на ширинах 1440, 1024, 768, 390 и 360 пикселей;
- на телефоне видны границы недели и доступен быстрый переход между днями без превращения приложения в длинный общий список;
- интерактивные элементы имеют удобный размер, текст не обрезается критически, горизонтальная прокрутка используется только там, где она осознанно поддерживает недельный обзор;
- все критерии из `concept.md` пройдены на вымышленных данных;
- сборка, модульные и сквозные тесты проходят без ошибок;
- проверка репозитория не находит реальных данных или секретов.

## 5. Стратегия хранения данных в localStorage

Для первой версии рекомендуется хранить состояние одним версионированным документом под ключом `weekflow:v1:state`:

```ts
type StoredState = {
  schemaVersion: 1;
  projects: Project[];
  tasks: Task[];
  preferences: {
    weekStartsOn: 1;
  };
};
```

Принципы хранения:

- слой хранения изолирован от компонентов интерфейса;
- чтение выполняется один раз при запуске, запись — после подтвержденного изменения состояния;
- перед использованием данные проходят разбор и проверку структуры;
- записи выполняются целиком, чтобы проекты и задачи не рассинхронизировались между несколькими ключами;
- формат имеет `schemaVersion`; будущие изменения выполняются через явные миграции;
- демоданные добавляются только при первом запуске, когда ключ отсутствует, и не восстанавливаются поверх пользовательских данных автоматически;
- очистка данных и восстановление демоданных требуют явного подтверждения;
- ошибка чтения или нехватка места показывается пользователю; приложение не должно молча сообщать об успешном сохранении;
- приложение не использует сервер, аналитику или внешнюю отправку содержимого задач.

`localStorage` подходит для MVP, но имеет ограничения: данные связаны с конкретным браузером, не синхронизируются, могут быть очищены пользователем и не являются резервной копией. Это должно быть явно объяснено в интерфейсе.

## 6. Адаптация интерфейса для Mac и телефона

### Mac

- неделя показывается как единая семидневная сетка;
- зона «Сегодня» и быстрое добавление располагаются рядом с сеткой или над ней, не сокращая ее до нечитаемого размера;
- многодневные полосы строятся на общей координатной сетке, чтобы непрерывность была очевидна;
- поддерживаются мышь, трекпад и клавиатура, видимый фокус и предсказуемый порядок перехода по элементам.

### Телефон

- сохраняется контекст всей недели: компактные заголовки семи дней и индикаторы загрузки остаются видимыми;
- подробности дней могут располагаться в горизонтально прокручиваемой недельной области или раскрываться для выбранного дня;
- многодневная полоса должна визуально сохранять начало, продолжение и завершение, а не превращаться в дублированные карточки;
- «Сегодня» и быстрое добавление размещаются до подробного списка или открываются компактно, чтобы основное действие было доступно одним касанием;
- боковые панели превращаются в полноэкранные нижние или модальные панели;
- сенсорные цели имеют размер не менее 44 × 44 пикселей, поля не вызывают нежелательное масштабирование, критические действия не зависят от наведения мыши.

Конкретный мобильный вариант нужно проверить прототипом до реализации сложной логики полос.

## 7. Работа с демонстрационными данными

- использовать только явно вымышленные проекты, например «Северный ветер», «Личный фокус» и «Курс по дизайну»;
- использовать нейтральные вымышленные задачи без имен, адресов, контактов, договоров и реальных сумм;
- подготовить сценарий, который демонстрирует однодневную задачу, многодневную полосу, просрочку, высокий приоритет, завершенную задачу и входящую запись;
- хранить демоданные отдельно от логики приложения в явно названном модуле;
- создавать новый набор демоданных при первом запуске, чтобы пользовательские изменения не мутировали исходный шаблон;
- не использовать в тестах, документации и скриншотах копии реальных рабочих данных;
- восстановление демоданных выполнять только после предупреждения, что текущие локальные данные будут заменены.

## 8. Приватность и проверка перед публикацией

До каждого публичного размещения необходимо:

1. Просмотреть текущие изменения и всю историю коммитов на наличие реальных имен, задач, названий клиентов, контактов и внутренних ссылок.
2. Выполнить поиск по известным чувствительным словам, адресам электронной почты, доменам, токенам и типичным форматам секретов.
3. Проверить демоданные, тесты, снимки экрана, документацию, сообщения об ошибках и метаданные файлов.
4. Убедиться, что пользовательские данные существуют только в браузере и не экспортируются в файлы репозитория.
5. Проверить, что в коде нет аналитики, сетевых запросов или сторонних SDK, способных получить содержимое задач.
6. Убедиться, что локальные служебные файлы, переменные окружения и будущие секреты исключены из публикации.
7. При обнаружении реальных данных не ограничиваться удалением из последнего состояния: до публикации очистить их из всей истории безопасным согласованным способом.

Проверка должна выполняться на вымышленных данных. Публикация на GitHub производится только по отдельному явному запросу пользователя.

## 9. План Git-коммитов

Коммиты создаются после успешной приемки крупных этапов, только по явному разрешению пользователя. Рекомендуемая последовательность:

1. `docs: define WeekFlow concept and MVP plan`
2. `chore: scaffold React TypeScript app`
3. `feat: add domain model and local storage`
4. `feat: build weekly overview and multi-day tasks`
5. `feat: add today view and task editing`
6. `feat: add quick capture and inbox workflow`
7. `feat: add project management and visual system`
8. `test: cover core planning workflows`
9. `fix: polish responsive layout and accessibility`
10. `docs: document local data and privacy safeguards`

Перед каждым коммитом:

- проверить состав изменений и не включать посторонние файлы;
- запустить относящиеся к этапу проверки;
- убедиться, что используются только вымышленные данные;
- не смешивать исправления разных этапов без необходимости.

Настройки Git не меняются. Отправка ветки и создание репозитория или pull request не входят в этот план без отдельного поручения.
